# AI駆動型 ネットワーク設計検証の自律エージェントシステム

Cisco CML環境における「AI駆動型 ネットワーク設計・検証自律エージェントシステム」の全体設計の概要をまとめます。

## システム全体像

本システムは、人間の曖昧な要件定義（自然言語）からスタートし、設計・デプロイ・検証・デバッグのサイクルをAIエージェントが自律的に回し続けるシステムです。

```
[ 人間 (インプット: 要件) ]
       │
       ▼
 ┌────────────────────────────────────────────────────────┐
 │ 🤖 LangGraph (オーケストレーション)                     │
 │                                                        │
 │  ┌───────────────────────┐   (エラーログ / 再設計依頼)   │
 │  │ 1. 設計エージェント    │ ◀─────────────────────────┐ │
 │  │   (LLM: CCIE頭脳)     │                           │ │
 │  └──────────┬────────────┘                           │ │
 │             │ (トポロジーYAML / 各機器コンフィグ)     │ │
 │             ▼                                        │ │
 │  ┌───────────────────────┐                           │ │
 │  │ 2. 検証エージェント    │ ──────────────────────────┘ │
 │  │   (LLM: テストエンジニア)│                               │
 │  └──────────┬────────────┘                               │
 └─────────────┼────────────────────────────────────────────┘
               │ (API / テスト実行コマンド)
               ▼
 ┌────────────────────────────────────────────────────────┐
 │ 🛠️ ネットワーク操作・自動化層                             │
 │   - virl2_client (CML操作)                              │
 │   - pyATS / Genie (コンフィグ投入・ログパース)         │
 └─────────────┼────────────────────────────────────────────┘
               │ (仮想環境の制御・通信テスト)
               ▼
 ┌────────────────────────────────────────────────────────┐
 │ 💻 実証環境 (Cisco CML)                                │
 │   - IOS-XE, NX-OS などの仮想ルータ/スイッチ             │
 └────────────────────────────────────────────────────────┘
```


## 主要コンポーネントと役割

システムは大きく4つの階層・コンポーネントで構成されます。

1. 状態管理（State）

役割: エージェント間で共有される「共通の記憶」

保持データ: 人間の要件、CMLトポロジー（YAML）、各機器のコンフィグ、現在のテスト結果（PASS/FAIL）、失敗時のエラーログ。

2. 設計エージェント（Architect Agent）

役割: 要件の解釈と、それに基づくネットワークアーキテクチャの生成。

出力:

CMLに読み込ませる接続定義（YAMLファイル）

各機器（IOS-XE等）に流し込む初期設定テキスト（IP, 凡例, ルーティング等）

特殊トリガー: 検証エージェントから「エラーログ」が戻ってきた場合、その原因を特定し、コンフィグやトポロジーを修正して再出力する。

3. 検証エージェント（Validation Agent）

役割: 構築された環境のテスト計画の立案と評価。

出力: テスト実行結果（成功 or 原因分析レポート）

動作フロー:

設計データをCMLへデプロイする（ツールを呼び出す）。

要件に必要なテスト（Ping, ルーティングテーブル確認など）を自ら判断して実行する。

結果をパースし、不合格なら「なぜ失敗したか」の推論を添えて設計エージェントに差し戻す（ループ）。

4. 自動化・環境操作ツール（Tools）

AIの「手足」となるPython関数群です。LLMはこれらを必要に応じて呼び出します。

CML操作（virl2_client）: ラボの新規作成、起動・停止、障害シミュレーション（リンクダウン）。

機器操作・テスト（pyATS / Genie）: コンフィグの流し込み、および show コマンドの結果をJSON構造化データとしてAIへ返却。

## 自律運用のワークフロー（処理の流れ）

【要件入力】 人間が「拠点をOSPFで冗長化し、VLAN10と20を隔離して」と入力。

【トポロジー＆設計】 設計AIがCML用YAMLとルータコンフィグを生成。

【デプロイ】 検証AIがツールを使い、CML上にラボを立ち上げコンフィグを自動注入。

【自動テスト】 検証AIがpyATSで状態を確認。「OSPFネイバーが上がっていない」ことを検知。

【推論・修正ループ】 検証AIが「ルータAとBのエリア番号がミスマッチしている」とログから推論し、設計AIにフィードバック。設計AIが修正コンフィグを作成し、手順3へ戻る。

【完了・承認】 すべてのテストが「PASS」になったら、人間に「検証成功」のレポートを出力して終了。

## 期待される効果

人間のSEの役割の変化: 「設計書の作成やコマンドラインへの打ち込み」という実作業から解放され、最初の「要件定義」と最後の「検証結果レポートの承認（Human-in-the-Loop）」だけに集中できるようになります。

検証の網羅性向上: 人間がやると手間の最大の難関である「片系リンク断の障害試験」なども、AIエージェントならCML APIを使って何パターンでも夜間に自動で回し切ることができます。
